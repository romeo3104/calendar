# calendar

GitHub Pages で公開する年カレンダーと、当日から参照できる世界経済サマリーを生成する静的サイトです。

このリポジトリは、Python スクリプトで静的 HTML を生成し、GitHub Actions で自動ビルドして GitHub Pages へ公開する構成です。

## 1. 目的

このプログラムの目的は次のとおりです。

- 日本の祝日を含む年カレンダーを静的サイトとして公開する
- 1990年から2050年までの年ページを生成する
- 現在年を起点に、前後の年へ簡単に移動できるようにする
- 当日セルから、NY市場クローズ後の世界経済サマリーを参照できるようにする
- GitHub Pages だけで運用できるようにする
- ブラウザ側の動的 API 呼び出しに依存せず、ビルド時に情報を静的生成する
- PC とスマートフォンの両方で閲覧しやすいレスポンシブ表示を行う

## 2. 全体構成

主要ファイルは以下です。

- `scripts/build_calendar.py`
  - カレンダー本体を生成する
- `scripts/build_world_summary.py`
  - 世界経済サマリーを生成する
- `.github/workflows/deploy.yml`
  - GitHub Actions によるビルドと Pages デプロイを行う
- `src/style.css`
  - カレンダー画面とサマリー画面の共通スタイル
- `src/favicon.svg`
  - ファビコン
- `dist/`
  - ビルド成果物の出力先
- `dist/summary/`
  - 世界経済サマリーの出力先

## 3. カレンダー機能仕様

### 3-1. 生成対象年

- 生成対象は **1990年から2050年まで** です
- 各年ごとに `YYYY.html` を生成します
- `index.html` は実行日の現在年を表示するページになります

生成例:

- `1990.html`
- `2026.html`
- `2050.html`
- `index.html`

### 3-2. 年タイトル

年タイトルは以下の形式です。

- `2026年（令和8年）午年[うま年]カレンダー`

表示内容:

- 西暦
- 和暦
- 干支
- 「カレンダー」という名称

### 3-3. 月タイトル

月タイトルは以下の形式です。

- `January (睦月)`
- `February (如月)`
- `March (弥生)`

表示内容:

- 英語の月名
- 和風月名

### 3-4. ナビゲーション

年ナビゲーションは以下を持ちます。

- `前`
  - 1年前のページへ移動
- `現在表示中の年`
  - `2026年` のような年ラベルを表示
- `次`
  - 1年後のページへ移動
- `今`
  - 実行日時点の現在年ページへ移動

境界条件:

- 1990年ページでは `前` は無効表示
- 2050年ページでは `次` は無効表示
- 現在年を表示中は `今` を無効表示

### 3-5. 月パネルのサイズ統一

月ごとのパネルサイズ差をなくすため、各月は常に **6週間分** を描画します。

そのため:

- 4週で収まる月でも空セルを追加する
- 5週の月でも6週目を確保する
- すべての月パネルの高さを揃える

### 3-6. 当日表示

実行日の当日セルには以下を適用します。

- 背景色の強調
- 枠線の強調
- 太字表示

さらに、当日セルの日付数字はリンク化されます。

リンク先:

- `summary/latest.html`

### 3-7. 祝日表示

日本の祝日をセル内に表示します。

表示内容:

- 日付
- 祝日名

対象:

- 元日
- 成人の日
- 建国記念の日
- 天皇誕生日
- 春分の日
- 昭和の日
- 憲法記念日
- みどりの日
- こどもの日
- 海の日
- 山の日
- 敬老の日
- 秋分の日
- スポーツの日または体育の日
- 文化の日
- 勤労感謝の日
- 振替休日
- 国民の休日

補足:

- 年代により祝日法の変更があるため、年によって固定日とハッピーマンデーの扱いが変わります
- 春分日と秋分日は近似式で算出しています
- 1949年以降の祝日法ベースを前提にしています
- 生成対象年は 1990年から2050年なので、この範囲内での運用を想定しています

## 4. 世界経済サマリー機能仕様

### 4-1. 概要

当日セルをクリックすると、最新の世界経済サマリーページを表示します。

表示先:

- `summary/latest.html`

このページは GitHub Actions で事前生成される静的 HTML です。

### 4-2. 生成タイミング

世界経済サマリーは以下のタイミングで生成されます。

- `push`
- `workflow_dispatch`
- GitHub Actions の定期実行

定期実行の考え方:

- NY市場クローズ10分後を想定
- 夏時間と冬時間の違いを吸収するため、workflow は 2 本の UTC 時刻を設定
- 実際に生成するかどうかは Python 側で `America/New_York` の時刻を見て判定

### 4-3. サマリー対象データ

市場データは次のカテゴリを対象にします。

- 株価
  - S&P 500
  - NASDAQ
  - ダウ
  - 日経225
- 先物
  - S&P 500 先物
  - NASDAQ100 先物
  - NYダウ先物
- 債券・金利
  - 米3か月
  - 米5年
  - 米10年
  - 米30年
- 為替
  - ドル円
  - ユーロドル
  - ドル指数
- コモディティ
  - WTI原油
  - ブレント原油
  - 金
  - 銀
  - 銅
- 暗号資産
  - Bitcoin
  - Ethereum

### 4-4. ニュース対象

ニュースは以下の系統を対象にします。

- Reuters 系
- Bloomberg 系

現在の実装では、各媒体ドメインを条件にしたニュース RSS を使用して見出しを取得します。

### 4-5. サマリー画面の表示内容

サマリー画面には以下を表示します。

- 生成日時
  - JST
  - New York
- 概況
  - 株式
  - 金利・為替
  - コモディティ
  - 暗号資産
- 主要指標カード
- カテゴリ別テーブル
- ニュース見出し一覧
- カレンダーへ戻るボタン

### 4-6. 生成ファイル

サマリーは以下を出力します。

- `dist/summary/latest.html`
- `dist/summary/latest.json`
- `dist/summary/YYYY-MM-DD.html`
- `dist/summary/YYYY-MM-DD.json`

用途:

- `latest.html`
  - 当日セルからの参照先
- 日付付きファイル
  - 生成履歴の保存用

### 4-7. データ取得失敗時の挙動

市場データまたはニュース取得に失敗しても、可能な範囲でページ生成を継続します。

挙動:

- 個別銘柄の取得失敗は `取得失敗` などの表示
- サマリー自体の HTML は生成継続
- 例外はログに出力

## 5. デザイン仕様

### 5-1. テーマ

テーマは **白系** です。

主な方針:

- 背景は淡い白系
- パネルは白背景
- 罫線は薄いグレー系
- 当月は青枠で強調
- 日曜は赤
- 土曜は青
- 祝日は赤寄り
- 当日は黄系で強調

### 5-2. レイアウト

カレンダー画面:

- 3カラムグリッド
- 年ナビゲーションを上部に表示
- 各月パネルは高さを揃える

サマリー画面:

- 概況セクション
- 指標カードグリッド
- カテゴリ別テーブル
- ニュース一覧

### 5-3. レスポンシブ対応

このプログラムはレスポンシブ表示に対応します。

#### カレンダー画面

- 幅が十分ある画面では 3 カラム表示
- 中間幅では 2 カラム表示
- スマートフォン幅では 1 カラム表示
- 年ナビゲーションは折り返し可能
- 月パネルは狭い画面では縦方向に並ぶ

#### サマリー画面

- 指標カードは広い画面で複数列表示
- 中間幅では 2 列表示
- スマートフォン幅では 1 列表示
- テーブルは横スクロール可能
- 見出しとナビゲーションは縦積みに切り替わる

#### 実装上の考え方

- `@media (max-width: 1200px)` で中間幅レイアウトへ切替
- `@media (max-width: 760px)` でスマートフォン向けレイアウトへ切替
- 固定幅ではなく `grid` と `minmax()` を使って列数を調整
- スマートフォンでは余白を縮小し、セル高さもやや小さくする

### 5-4. ファビコン

対応ファイル:

- `src/favicon.svg`
- `src/favicon.ico`
- `src/favicon.png`

存在するファイルのみ `<head>` に埋め込み、`dist/` へコピーします。

- `push`

1. リポジトリ取得

## 9. 既知の制約

- 市場データは外部サービス取得なので、取得失敗や遅延の可能性があります
- 春分日・秋分日は近似計算です
- 過去年の祝日制度変更は、主要ルールは反映していますが厳密な全例外までは未検証です
- GitHub Pages は静的配信なので、クリック時にその場で外部 API をリアルタイム呼び出しする構成ではありません
- ニュースはビルド時点の取得内容です

## 10. 想定利用方法

このプログラムは、次のような用途を想定しています。

- 年カレンダーを長期で参照する
- 過去年や未来年の祝日配置を確認する
- 当日からNY市場クローズ後の世界経済サマリーを見る
- GitHub Pages 上で軽量に公開・運用する
- PC とスマートフォンの両方から参照する
